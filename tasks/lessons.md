# Lessons

- Keep content-only tasks isolated from config and layout work when the repo is being bootstrapped.
- Check the existing tree before adding files so new content matches the actual project state.
- When designing a personal blog, avoid symmetrical card-grid layouts that feel like SaaS marketing by default.
- Establish authorial voice, information hierarchy, and a distinctive publication metaphor before polishing colors or hover states.
- For this blog, default to plain, ordered, content-first presentation; avoid glass cards, ornamental gradients, oversized display serif text, and rounded-pill navigation unless explicitly requested.
- Treat the blog archive as a directory product, not a landing page: prioritize list scanning, strong hierarchy, hover preview, and search over fragmented content blocks.
- Authoring workflow is part of the product surface: editor UX, Markdown ergonomics, import flows, and publishing steps must be designed together with the public pages.
- Do not call the redesign complete until light/dark theme sync, mobile/tablet/desktop responsiveness, and article utilities like code/math copy actions have been verified.
